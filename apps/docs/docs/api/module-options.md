# Module Options

## AuthModule.forRoot

Synchronous configuration with inline options:

```typescript
AuthModule.forRoot(options: AuthModuleOptions)
```

## AuthModule.forRootAsync

Asynchronous configuration (recommended for production):

```typescript
AuthModule.forRootAsync({
  imports: [...],
  inject: [...],
  useFactory: (...deps) => ({ ... }),
})
```

Both entry points validate the options at startup (`validateAuthModuleOptions`)
and throw immediately if a secret is missing, shorter than 32 characters, the
access/refresh secrets are identical, a duration is malformed, or the token
store is missing. This turns misconfiguration into a boot-time error instead of
a runtime auth bug.

## AuthModuleOptions

```typescript
interface AuthModuleOptions {
  /** Path to keycloak-realm.json generated by `auth-cli init` */
  keycloakConfigPath?: string;

  /** Inline Keycloak config (alternative to keycloakConfigPath) */
  keycloak?: KeycloakConfig;

  /** JWT signing configuration (secrets must be >= 32 chars and distinct) */
  jwt: JwtConfig;

  /** Email sender configuration */
  email: EmailConfig;

  /** Token store implementation (PrismaTokenStore, Redis, etc.) */
  tokenStore: ITokenStore;

  /** Base URL for email links (default: https://example.com) */
  baseUrl?: string;

  /** Secure cookie flag (default: true when NODE_ENV=production, else false) */
  cookieSecure?: boolean;

  /** Cookie domain (optional) */
  cookieDomain?: string;

  /**
   * Require email verification before first login.
   * When true, register creates the account with emailVerified=false, sends a
   * verification email (needs SMTP configured in the Keycloak realm), and login
   * fails with 403 EMAIL_NOT_VERIFIED until the user verifies. Default: false.
   */
  requireEmailVerification?: boolean;

  /** Rate limiting (integrated via @nestjs/throttler — see the rate-limiting recipe) */
  rateLimit?: {
    enabled?: boolean; // turn throttling off entirely (default: true)
    limit?: number;    // requests per window (default: 10)
    ttl?: number;      // window in seconds (default: 60)
  };

  /** Health check endpoint */
  healthCheck?: {
    enabled?: boolean; // GET /auth/health (default: true; 404 when false)
  };

  /** Account lockout after repeated failed logins */
  lockout?: {
    maxAttempts?: number; // default: 5
    duration?: number;    // lockout window in ms (default: 15 min)
  };

  /** CSRF double-submit-cookie protection */
  csrf?: {
    enabled?: boolean;    // enforce on mutating requests (default: false)
    cookieName?: string;  // default: csrf-token
    headerName?: string;  // default: x-csrf-token
  };

  /** Opt out of the app-wide providers the module registers */
  globals?: {
    authGuard?: boolean;      // register JwtAuthGuard as APP_GUARD (default: true)
    validationPipe?: boolean; // register a whitelist ValidationPipe (default: true)
  };
}
```

## CSRF protection

When `csrf.enabled` is `true`, the module registers `CsrfGuard` app-wide and
exposes `GET /auth/csrf`. Clients must first fetch the CSRF token, then echo it
back in the configured header on every mutating request:

```typescript
// 1. Fetch the token (also sets a non-httpOnly `csrf-token` cookie)
const { csrfToken } = await fetch("/auth/csrf").then((r) => r.json());

// 2. Send it back on POST/PUT/PATCH/DELETE
await fetch("/auth/login", {
  method: "POST",
  headers: { "x-csrf-token": csrfToken, "content-type": "application/json" },
  body: JSON.stringify({ email, password }),
});
```

The cookie and header are compared with a timing-safe equality check. Safe
methods (GET/HEAD/OPTIONS) are never challenged.

## Exports

The module exports (injectable in the consumer app):

- `AUTH_MODULE_OPTIONS`, `TOKEN_STORE`, `AUTH_PROVIDER` — DI tokens
- All use cases (`LoginUseCase`, `RefreshTokenUseCase`, …)
- `AuthEventBus` — subscribe to auth events (see [Events](./events.md))
- `JwtTokenService`, `ReactEmailRenderer`
- `JwtAuthGuard`, `OptionalAuthGuard`, `CsrfGuard`, `ThrottlerGuard`
